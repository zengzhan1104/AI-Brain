# lint 技能：知识图谱健康巡检

## 核心目标
将软件工程中的“静态代码分析”引入知识管理。定期运行此 skill，找出知识库长期演进中产生的：死链、孤岛、未同步索引、认知冲突。

## 触发条件
- 用户输入 `/lint`
- 用户询问“我的知识库健康状况如何”
- 用户要求“检查知识库状态”或“检查健康”

## 知识库路径
- 使用 Glob 工具，**path 参数设为 vault 根目录（`../`）**，pattern 为 `*/wiki/` 以覆盖所有子库

## 巡检流水线

### 第 1 步：索引一致性检查
1. 读取 vault 根 `../index.md`（统一索引）全部内容
2. 扫描所有 `*/wiki/` 下所有 `.md` 文件（递归子目录，排除各子库的 `log.md`）
3. 提取 index.md 中注册的所有双链链接 `[[页面名称]]`
4. 比对：找出已注册但文件不存在的条目 OR 文件存在但未注册的页面

### 第 2 步：双向链接健康检查
1. 扫描所有 `*/wiki/` 下所有 `.md` 文件（递归子目录，排除各子库的 `log.md`），提取所有 `[[双链]]` 格式的出链
2. 扫描每个页面的 YAML frontmatter，提取 `aliases` 字段，构建 `{别名 → 实际文件名}` 映射表
3. 对每条出链的目标，在文件列表和别名映射表中查找 → 均不存在则标记为**死链**
4. 统计各页面被引用次数（排除 self-reference）。`wiki/log.md` 已在 glob 过滤时排除，不参与统计。
5. 从未被任何其他页面引用的页面（且不在排除列表中） → **孤儿页面**

### 第 3 步：必填区域检查
1. 扫描所有 `*/wiki/` 下所有 `.md` 文件（递归子目录，排除各子库的 `log.md`）
2. 检查每个页面是否包含 `## 关联连接` 标题
3. 缺失的页面 → 标记为**黄灯项**："缺少必填的关联连接区域"

### 第 4 步：认知冲突审查
1. 全局搜索所有 `.md` 文件中包含 `## 知识冲突` 的页面
2. 提取每个冲突的简要描述（冲突双方是什么）
3. 统计带未解决冲突的页面（认知技术债）

### 第 5 步：概念/方法语义去重检查

1. 扫描所有 `*/wiki/concepts/` 和 `*/wiki/methods/` 下所有 `.md` 文件，提取每个页面的：
   - `title`（YAML frontmatter）
   - `aliases`（YAML frontmatter，若存在）
   - `## 定义` 区域的首段（用于后续合并决策时的上下文展示）
   - 所属子库（从文件路径中提取）
2. 构建全局 `{别名 → [文件名列表]}` 映射表（多个页面可能共享同一别名，含跨子库场景）
3. **红灯检测（高置信度）**：遍历每个页面，若其 `title` 出现在另一个页面的 `aliases` 中，且两个页面文件均独立存在 → 标记为"高置信度重复"（跨子库同名同样适用）
4. **黄灯检测（中置信度）**：若同一别名在别名映射表中对应多个页面 → 标记为"中置信度重叠"（特别标注跨子库重叠，因二者分属不同子库，可能编码了不同的视角或定义偏重）
5. **建议检测（低置信度）**：提取每个标题的实义词（去停用词、去标点、大小写归一化），若两页共享 ≥2 个关键词且标题非完全相同 → 标记为"建议审阅"（跨子库时额外标注来源子库）
6. **仅报告，不自动处理**——合并决策完全交由用户逐对确认

### 第 6 步：数学公式规范性检查

扫描所有 `*/wiki/` 下所有 `.md` 文件，检测以下 7 类公式问题：

#### 6.1 未闭合的块级公式分隔符 `$$`

- 规则：统计每个页面中 `$$` 出现次数，若为奇数 → 标记为**红灯**（未闭合）
- 排除：``` 代码块和 ` 行内代码中的 `$$`
- 实现：逐行扫描，维护"当前是否在数学块内"状态机

#### 6.2 行内公式 `$` 疑似未闭合

- 规则：单行中出现奇数个 `$`（排除 `$$`），或 `$` 跨越段落边界
- 排除：代码块内的 `$`；行首/行尾孤立的 `$`
- 标记为**红灯**

#### 6.3 Unicode 数学字符未转为 LaTeX 命令

检测以下高频 Unicode 字符是否直接出现在正文中（未包裹在代码块内），应替换为 LaTeX 等价命令：

| Unicode 字符 | 应使用 | 含义 |
|-------------|--------|------|
| `μ` | `\mu` | 微/动力粘度 |
| `β` | `\beta` | 贝塔 |
| `Ω` | `\Omega` | 欧姆/大 omega |
| `·` | `\cdot` | 乘点 |
| `×` | `\times` | 乘号 |
| `±` | `\pm` | 正负号 |
| `≤` | `\leq` | 小于等于 |
| `≥` | `\geq` | 大于等于 |
| `α` | `\alpha` | 阿尔法 |
| `γ` | `\gamma` | 伽马 |
| `Δ` | `\Delta` | 德尔塔 |
| `ε` | `\epsilon` | 伊普西龙 |
| `θ` | `\theta` | 西塔 |
| `λ` | `\lambda` | 拉姆达 |
| `π` | `\pi` | 派 |
| `ρ` | `\rho` | 密度 |
| `σ` | `\sigma` | 西格玛 |
| `τ` | `\tau` | 陶 |
| `ω` | `\omega` | 小 omega |
| `η` | `\eta` | 伊塔 |

- 仅标记位于正文段落中的字符（排除代码块和 YAML frontmatter）
- 每个页面标记为**黄灯**，列出具体字符和位置（行号）

#### 6.4 Unicode 上下标未使用 LaTeX 语法

| Unicode | 应使用 | 
|---------|--------|
| `²` | `^2` |
| `³` | `^3` |
| `₁` `₂` `₃` `₄` | `_1` `_2` `_3` `_4` |
| `°` (用于温度) | `^\circ` |

- 规则同上：排除代码块和 frontmatter
- 标记为**黄灯**

#### 6.5 下标变量缺少 `$` 包裹

- 规则：检测形如 `T_w`、`Nu_eff`、`R_{total}` 的下标变量（大写字母/希腊字母起头 + `_` + 下标，或单小写字母 + `_` + 下标），若未包裹在 `$...$` 或 `$$...$$` 数学模式内 → 标记为**黄灯**
- 排除：
  - 已在 `$...$` 或 `$$...$$` 内的
  - 代码块内
  - 常见拉丁短语（et_al, in_situ 等）
  - Wiki 双链 `[[...]]` 内的文件名
  - 单希腊字母作为独立符号与后续变量连写的情况（如 `φk_f` 实为 `φ·k_f`）
- 示例：`T_w` → 应修正为 `$T_w$`；`T_max` → 应修正为 `$T_{max}$`

#### 6.6 行内公式中多字符下标缺花括号

- 规则：在已有的 `$...$` 行内数学块内，检测形如 `$T_sat$` 的模式（变量单字母 + 多字符下标未用 `{}` 包裹）
- 示例：`$T_sat$` → 应修正为 `$T_{sat}$`；`$ΔT_sub$` → 应修正为 `$ΔT_{sub}$`
- 标记为**黄灯**

#### 6.7 显示数学块 `$$...$$` 内存在多余 `$`

- 规则：`$$...$$` 块级公式内部已经是数学模式，不应再出现行内 `$`。检测每个 `$$...$$` 块内部是否包含单 `$` 符号
- 标记为**红灯**（公式渲染错误）
- 示例：`$$\frac{$T_5$ - $T_2$}{\Delta Z}$$` → 块内 `$T_5$` 和 `$T_2$` 的 `$` 应移除

**实现方式**：用 Python 脚本批处理，不可逐页手动检查（页面量大且易遗漏）。脚本路径：`.claude/skills/lint/check_formulas.py`，由 lint 技能在步骤 6 中调用。

### 第 7 步：收件箱积压检查
- `raw/` 收件箱积压检查不在本 skill 范围内，由 `/ingest` 技能负责。

### 第 8 步：研究纲领同步检查
1. 读取 `../研究纲领.md`（vault 根），提取全文关键信息：
   - 核心研究领域（一级/二级方向、应用场景）
   - 当前关注的问题
   - 方法工具箱
   - 研究对象/体系

2. 扫描 `../研究日志/` 下最近 2 个月的日志文件，提取所有条目中的：
   - 文献阅读主题（按标题和关键词归类）
   - 实验/仿真/测试活动涉及的方法
   - 用户记录的新想法、方向调整表述
   - 反复出现但纲领未覆盖的新关键词或主题

3. 语义对比分析，识别以下信号：

   | 信号类型 | 判断标准 | 级别 |
   |---------|---------|------|
   | 纲领覆盖缺口 | 日志中反复出现纲领未提及的新主题/材料/方法 | 黄灯 |
   | 纲领冗余 | 纲领中列出的方向，近 2 月日志中零相关活动 | 黄灯 |
   | 关注点细化 | 纲领某方向在日志中表现出更细粒度的分化（如二级方向下出现子方向） | 建议 |

4. 单一事件不构成信号——须判断是否为反复出现或用户明确标注为新方向的活动。

5. 若以上对比检测到纲领覆盖缺口、冗余或细化信号，继续扫描所有 `*/wiki/sources/` 下所有精读页面，读取每篇的"与我的研究关联"字段（含「研究概要」中的一句话概括和「我的判断」中的四维展开），对照当前纲领重新评估：

   | 场景 | 判断标准 | 影响 |
   |------|---------|------|
   | 可能低估 | 论文主题与纲领新增方向高度相关，但旧关联判断未体现 | 建议重新评估 |
   | 可能过时 | 旧关联判断聚焦于纲领已删除/降级的方向 | 建议标注或更新 |
   | 细化不足 | 旧判断对纲领中已细化的方向描述过于笼统 | 建议细化 |

6. 将受影响的页面按影响类型归类，在报告中列出（含具体原因，如"纲领新增了 X 方向，该论文研究主题与之高度重合"）。

7. 若纲领无变化（对比方式：检查 `../研究纲领.md` 的 `updated` 日期是否晚于 `wiki/log.md` 中上次 lint 的日期；若 log.md 无历史 lint 记录则视为首次运行，执行 5-6），跳过 5-6。

## 报告输出规范

扫描完成后，输出结构化报告，严格遵循以下格式：

```markdown
## 知识库健康体检报告 — YYYY-MM-DD

### 绿灯项
- 全部检查通过的项（如"所有 N 个页面均含关联连接"、"索引入口全部有效"、"无死链"等）。若某项检查零问题，在此列出。
- **研究纲领与实际研究一致**（无覆盖缺口、冗余或细化信号时）

### 黄灯项
- **发现 N 个孤儿页面**：[列表] - 建议添加关联或分类
- **发现 N 个未同步索引**：[列表] - 文件存在但未在 index.md 注册
- **发现 N 个页面缺少关联连接**：[列表] - 必填区域缺失
- **研究纲领可能需更新**：[覆盖缺口/冗余信号] — 依据：[日志日期与条目摘要]
- **研究纲领关注点可细化**（建议）：[细化信号] — 仅为建议，非必须操作
- **发现 N 篇精读关联判断可能过时**：[页面列表与影响类型] — 纲领已变更，建议重新关联判断
- **公式格式（黄灯）**：
  - 发现 N 个页面存在 Unicode 数学字符（如 μ、Δ、·）未转为 LaTeX
  - 发现 N 个页面存在 Unicode 上下标（如 ²、³、₁）未使用 LaTeX
  - 发现 N 个页面存在下标变量缺少 `$` 包裹（如 `T_w` 应为 `$T_w$`）
  - 发现 N 个页面存在行内公式中多字符下标缺花括号（如 `$T_sat$` 应为 `$T_{sat}$`）
- **概念/方法去重**：
  - 🟡 中置信度重叠（N 对）：[[页面A]] 和 [[页面B]] 共享别名 `"xxx"`
  - 💡 建议审阅（N 对）：[[页面A]] 与 [[页面B]] 标题关键词重合（共享：xxx, yyy）

### 红灯项
- **发现 N 个死链**：[来源页面] → [[不存在的目标页面]]
- **存在 N 个未解决的知识冲突**：[页面名称]
- **公式错误（红灯）**：
  - 发现 N 个页面存在未闭合的 `$$` 块级公式
  - 发现 N 个页面存在疑似未闭合的行内 `$` 公式
  - 发现 N 个页面存在 `$$...$$` 显示数学块内有多余 `$`（块内不应出现行内 `$`）
- **概念/方法去重**：
  - 🔴 高置信度重复（N 对）：[[页面A]] ↔ [[页面B]]，互为别名，建议合并

### 下一步行动
1. 是否需要自动修复未同步索引？
2. 是否需要针对知识冲突进行重新推演？
3. 是否需要更新[[../研究纲领|研究纲领]]？（若有覆盖缺口或方向调整信号，建议同步更新以保持 ingest 关联判断准确）
4. 是否需要更新受影响的精读页面？（将对上述页面按当前纲领重跑四维比对，更新"与我的研究关联"字段）
5. 是否需要合并检测到的重复页面？（将逐对询问保留哪一方，合并后自动更新双向链接和索引）
```

## 修复：精读页面关联判断更新

> 当用户确认下一步行动第 4 项后执行。目标：以当前纲领重新评判已有精读页面，使关联判断与最新研究方向对齐。

### 流程

1. 逐一打开受影响的精读页面
2. 读取页面正文的「研究摘要」「研究背景」「研究结论」等区域（不重读 raw/ 原文），获取论文的核心信息
3. 对照当前 `../研究纲领.md`，按四维框架重新判断：

   | 维度 | 评估内容 |
   |------|---------|
   | 研究对象/体系 | 在当前纲领覆盖范围内吗？ |
   | 方法可迁移性 | 在当前纲领工具箱中或可补充吗？ |
   | 结论相关性 | 支撑/挑战/补充了当前纲领的哪些关注问题？ |
   | 知识空白 | 填补了当前纲领的哪个知识缺口？ |

4. 更新页面中两处字段：
   - 「研究概要」表格中的 `与我的研究关联`（一句话概括）
   - 「我的判断」中的 `与我的研究关联`（四维展开）

5. 在页面末尾追加变更记录（不修改其他区域）：

   ```markdown
   > **⤷ 关联判断更新于 YYYY-MM-DD**：纲领变更（简述变更内容），重新评估后调整了与我的研究关联判断。
   ```

6. 完成后更新 `wiki/log.md`。

## 修复：概念/方法页面合并

> 当用户确认下一步行动第 5 项后执行。逐对处理，每对单独询问。

### 流程

1. 对每对检测到的重复页面，展示：
   - 两页的 `title` 和 `aliases`
   - `## 定义` 区域首段
   - `## 来源追踪` 表格中的论文覆盖范围
2. 询问用户：
   > 检测到可合并的页面对：`[[页面A]]` ↔ `[[页面B]]`。请选择：
   > - **A)** 保留 `[[页面A]]`，将 `[[页面B]]` 内容合并入 A，删除 B
   > - **B)** 保留 `[[页面B]]`，将 `[[页面A]]` 内容合并入 B，删除 A
   > - **C)** 跳过此对，保留两者独立
   > - **D)** 跳过全部，不再处理任何合并
3. 用户选择 A 或 B 后执行合并：
   - 将被合并页的 `## 定义` 内容追加到保留页（仅写与前者的差异，不重复已有表述）
   - 将被合并页的「来源追踪」表行追加到保留页
   - 将被合并页的 `aliases` 合并入保留页（去重）
   - 将被合并页的 `sources` frontmatter 合并入保留页（去重）
   - 若被合并页有 `## 知识冲突` 内容，追加到保留页
4. **全局替换双链**：扫描所有 `*/wiki/` 下所有 `.md` 文件，将被删除页面的 `[[页面名]]` 替换为保留页面的 `[[页面名]]`
5. **更新 vault 根 `../index.md`（统一索引）**：移除已删除页面条目，必要时更新保留页面的描述
6. **删除被合并页面的文件**
7. 追加日志到 `wiki/log.md`：
   ```
   ## [YYYY-MM-DD] lint | 合并重复页面
   - **合并**: [[被合并页面]] → [[保留页面]]
   - **理由**: 两者互为别名/共享别名/标题关键词重合
   - **全局替换**: N 处双链引用已更新
   ```

## 硬约束
- **仅读扫描**：生成报告前，禁止修改、删除、重命名任何文件
- **手动确认**：报告后等待用户确认再执行修复
- **逐对询问**：合并操作每对单独向用户确认，不批量执行
- **静默日志**：修复完成后，在 wiki/log.md 追加，逐项列出修复内容：
  ```
  ## [YYYY-MM-DD] lint | 修复了 N 个问题
  - **死链修复**: [[来源页]] → [[目标页]]
  - **索引同步**: 新增 [[页面名]] 到 vault 根统一索引
  - **关联连接**: 为 [[页面名]] 补充关联连接
  - **页面合并**: [[被合并页]] → [[保留页]]（原因）
  ```

# 语言设定与核心角色 (Global Rules)
- **语言指令**：无论输入何种语言，你必须始终使用**简体中文**进行思考、回复和知识库的编写。
- **角色定义**：你正在维护一个 **LLM Wiki**（根据 [Karpathy 的规范](https://gist.github.com/karpathy/442a6bf555914893e9891c11519de94f))，你的任务是将碎片化的信息编译成结构化、高度相互链接的 Obsidian 知识库。
- **规则优先级**：本文件（项目级 CLAUDE.md）的规则与用户级 CLAUDE.md 冲突时，**以本文件为准**。具体说明：
  - **禁止并行 Agent**：**绝对禁止**使用 Agent 工具将任务拆分为多个子 agent 并行执行。所有工作需要深度思考和高质量输出（文献精读、概念提炼、代码编写等），必须在本会话中顺序逐项完成，确保每项任务获得完整的上下文关注。这适用于本项目的所有 skill 和工作流——包括但不限于 `/ingest`、`/lint`、`/query` 等。
  - **确认流程**：编写代码/修改文件前需复述理解并给出修改清单，等待用户确认后动手。
  - **编辑策略**：优先为每个独立概念/方法创建新页面，而非合并到已有页面——这与 wiki 的知识粒度设计一致，覆盖通用编码规范中的"优先编辑已有文件"。
  - **注释风格**：wiki 页面默认不加注释（遵循项目规范），代码注释遵循用户级 CLAUDE.md 的简洁原则。

# 多子库架构

本 vault 是一个**大知识库**，其下包含若干**子库**，每个子库对应一个研究领域：

```
Knowledge/                        ← vault 根（本目录）
├── .claude/                      ← 全局共享 skills（所有子库共用）
├── index.md                      ← 全库统一索引
├── log.md                        ← 全局操作日志
├── 研究纲领.md                   ← 跨子库统一研究纲领
├── 研究日志/                     ← 统一研究日志（按月分文件）
│   ├── index.md
│   └── 2026-05.md
├── _子库模板/                    ← 新建子库时复制此目录
│
├── 电力电子综述/                   ← 子库示例
│   ├── raw/                      ← 不可变层
│   ├── wiki/                     ← 编译输出层
│   └── assets/                   ← 媒体资产
│
└── （未来子库）/                  ← 按需扩展
```

**Vault 根关键文件**：
- `研究纲领.md`：全体子库共用的统一研究纲领。ingest 读取全文进行四维比对，lint 对照研究日志检测纲领同步状态
- `研究日志/`：统一研究日志（按月分文件），各子库活动按目录名标注区分
- `index.md`：Vault 全库统一索引
- `log.md`：全局操作日志

**子库间引用**：所有子库在同一 vault 内，Obsidian 双链可跨子库直接引用。示例：
- 同子库内引用：`[[微通道散热]]`
- 跨子库引用：`[[电力电子综述/wiki/concepts/微通道散热]]` 或 `[[微通道散热]]`（标题唯一时）
- 跨层引用（子库→vault 根）：`[[../研究纲领|研究纲领]]`、`[[../研究日志/index|研究日志]]`

# 核心目录与权限边界 (Immutability & Architecture)

对每个子库，你必须严格遵守以下文件操作权限，这是不可逾越的底线：

- `{子库}/raw/` (不可变层 - Immutable)：
  - **绝对只读**。存放原始素材、论文 Markdown。
  - **禁止修改或删除此目录下的任何文件**。它是事实的唯一真相来源。
- `{子库}/assets/` (媒体资产层)：
  - 存放图片。引用时使用 Obsidian 标准语法 `![[assets/xxx.png]]`。
- `{子库}/wiki/` (编译输出层 - You Own This)：
  - 你的专属工作区。在此创建、更新、提炼知识并解决矛盾。

# Wiki 核心文件契约 (The Wiki Schema)

当你在某个子库的 `wiki/` 中工作时，必须维护以下基石：

1. **vault 根 `index.md`（统一索引）**：每次向任何子库的 wiki 新增知识页后，同步更新此文件。格式：`[[子库名/wiki/分类/页面名]] — 一句话描述`，在 `## {分类}` → `### {子库名}` 分组下按年份/字母排列。
   - Concepts: 使用 TitleCase 命名（先翻译为简体中文）。
   - Sources/Syntheses: 使用 `{年份} - {标题}` 格式命名（中文论文用中文标题，英文论文保留英文标题）。
2. **`wiki/log.md` (子库操作日志)**：只能追加写入（Append-only）。
3. **内容分类**：
   - `/wiki/concepts/`：概念、框架、方法论、人物、公司、工具、产品
   - `/wiki/methods/`：测试、表征与数据处理方法及其证据链
   - `/wiki/sources/`：从 `raw/` 提炼出的论文精读
   - `/wiki/syntheses/`：跨文献综合分析、对比研究、领域综述
4. **强制双向链接**：每个 wiki 页面必须包含 `## 关联连接` 区域，使用 Obsidian 双链。
5. **矛盾处理原则**：新旧知识冲突时，新建 `## 知识冲突` 区块，两种说法都保留并做对比。

# 工作流指令说明 (Workflows / Skills)

所有子库共用全局 skills（位于 `.claude/skills/`）。在子库目录下启动 Claude Code 时：
- **子库内 skills**（`/ingest`、`/zotero-import`、`/mineru-convert`、`/research-log`）：操作范围自动限定在该子库内（使用相对路径）。
- **跨子库 skills**（`/query`、`/lint`）：操作范围为全 vault，通过 vault 根统一索引和 `*/wiki/` 跨子库 glob 实现。

- `/new-sublib`：从 `_子库模板/` 创建新子库
- `/ingest`：扫描当前子库 `raw/` 下未归档文件，提炼到 `wiki/`，更新 vault 根统一索引，归档至 `raw/archive/`
- `/research-log`：记录当前子库的研究日志
- `/query`：跨子库检索，三级降级（wiki → archive → 通用知识）
- `/lint`：全库健康巡检（死链、孤儿、索引同步、概念去重、公式规范、纲领同步）
- `/zotero-import`：从 Zotero 导入论文 PDF 到当前子库
- `/mineru-convert`：当前子库内 PDF → Markdown 转换
